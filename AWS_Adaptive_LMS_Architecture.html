SECTION 01

High-Level Architecture Overview & Data Flow

The architecture follows a serverless-first, event-driven pattern with three logical tiers inside a private VPC: an edge/delivery layer, a serverless compute layer, and a managed data layer. Every student request travels through CloudFront → API Gateway → Lambda → DynamoDB/S3, with AI/ML calls dispatched asynchronously to avoid adding latency to the user-facing path.

Why Each Service Was Chosen

Amazon CloudFront
Serves the React SPA and static assets from 450+ edge locations. Students in Nairobi and Mumbai get <50ms TTFB for course pages. Also terminates TLS and applies WAF rules before traffic ever reaches the VPC.
API Gateway (REST)
Single, versioned entry-point for all backend calls. Provides per-route throttling (prevents one student hammering the recommendation endpoint), request validation, and Cognito JWT authorisation out-of-the-box with zero code.
AWS Lambda
Zero-server compute that scales from 1 to 10,000 concurrent students automatically. Each function is a single-responsibility unit — the assignment grader never shares a runtime with the video transcriber. Cold starts are mitigated with Provisioned Concurrency on critical paths.
Amazon DynamoDB
Single-digit millisecond reads at any scale. The key-value + document model perfectly fits course catalogues (key = courseId), student progress (key = studentId + moduleId), and event logs (time-series). On-Demand pricing means 0 cost when students sleep.
Amazon Cognito
Handles registration, MFA, social login, and JWT issuance for 4,800 users without writing a single line of auth code. User Pools store student attributes; Identity Pools grant temporary AWS credentials for direct S3 video uploads from the browser.
AI/ML Triad
Transcribe converts video audio → timestamped JSON transcripts. Comprehend extracts sentiment and key-phrase entities from assignment text to auto-generate rubric feedback. Personalize trains a collaborative-filtering model to recommend the next optimal learning activity.

Step-by-Step Data Flow (Normal Request)

1
Student opens browser → DNS resolves to CloudFront distribution. CloudFront checks edge cache; on miss, fetches the React SPA bundle from S3 (origin). The SPA loads in <1.5s globally.
2
Login → Cognito Hosted UI authenticates credentials, issues ID Token + Access Token (JWT, 1h expiry). The React app stores tokens in memory (not localStorage) to prevent XSS token theft.
3
API call → The React app sends Authorization: Bearer <AccessToken> to the CloudFront distribution's /api/* behaviour, which forwards to API Gateway. API Gateway validates the JWT against the Cognito User Pool JWKS endpoint — no Lambda invocation needed for auth.
4
Lambda invocation → API Gateway routes the authenticated request to the correct Lambda function inside the Private App Subnet. The function reads/writes DynamoDB through a VPC Gateway Endpoint — traffic never leaves the AWS backbone.
5
AI/ML dispatch (async) → For video transcription and assignment analysis, Lambda publishes an event to an SQS queue, returns 202 Accepted immediately to the client, and a separate consumer Lambda processes the job asynchronously. This prevents 90-second Transcribe jobs from blocking the UI.
6
Notification → Once AI processing completes, SNS triggers the Notification Lambda which calls SES for personalised email and SNS Mobile Push for in-app alerts.
7
Monitoring → Every Lambda emits structured logs to CloudWatch. Custom metrics (quiz_score, video_completion_rate) feed a CloudWatch Dashboard. CloudTrail records every API call for compliance auditing. VPC Flow Logs capture network traffic at the ENI level.

SECTION 03

Complete System Workflows

Workflow A — Student Login

1
Student enters credentials on the React SPA hosted via CloudFront → S3.
2
SPA calls Cognito User Pool using the Amplify SDK. Cognito validates username/password against the User Pool.
3
Cognito returns IdToken (student attributes), AccessToken (API authorization), and RefreshToken (silent renewal every 60 min).
4
SPA calls GET /students/{studentId}/dashboard with Authorization: Bearer <AccessToken>.
5
API Gateway validates the JWT signature using the Cognito JWKS endpoint (cached 5 min) — zero Lambda invocation for auth validation.
6
StudentDashboardLambda queries DynamoDB: fetches current course enrolments, last activity timestamp, and pending assignments.
7
The same Lambda calls PersonalizeLambda synchronously to fetch the top-3 recommended next activities and injects them into the dashboard response.

Workflow B — Video Upload + Transcription Pipeline

1
Instructor uploads video → SPA requests a pre-signed S3 URL from POST /videos/upload-url.
2
VideoUploadLambda calls s3.createPresignedPost() scoped to s3://lms-videos-raw/{courseId}/{videoId}.mp4, 15-min expiry, max 2 GB.
3
Browser uploads directly to S3 (bypasses Lambda entirely — no 6 MB Lambda payload limit).
4
S3 Event Notification triggers TranscriptionDispatchLambda on s3:ObjectCreated:* in the raw/ prefix.
5
Lambda calls transcribe.startTranscriptionJob():
• MediaFileUri: S3 video URI
• LanguageCode: auto-detect (supports 37 languages for global students)
• Settings.ShowSpeakerLabels: true (multi-speaker detection)
6
Amazon Transcribe writes the completed JSON transcript to s3://lms-transcripts/{videoId}.json.
7
TranscriptProcessorLambda (triggered by the output S3 event) parses the transcript, extracts word-level timestamps, and generates chapter markers by detecting topic-shift sentences every ~5 minutes using Comprehend's Key Phrase detection.
8
Processed transcript + chapter markers written to DynamoDB Videos table. Student-facing search index updated. SNS sends instructor notification via SES email.

Workflow C — Assignment Submission + AI Feedback

1
Student submits assignment text via POST /assignments/{assignmentId}/submit.
2
AssignmentSubmitLambda writes submission to DynamoDB Submissions table, generates submissionId, publishes event to SQS:assignment-grading-queue, returns 202 Accepted immediately.
3
GradingLambda (SQS consumer, batch size=1) retrieves message. Calls comprehend.detectSentiment() + comprehend.detectKeyPhrases() + comprehend.classifyDocument() against a custom rubric classifier.
4
Comprehend returns structured JSON. GradingLambda maps rubric category scores → generates per-criterion feedback strings → computes a 0–100 score.
5
Feedback stored in DynamoDB. Student progress event (ASSIGNMENT_COMPLETED) published to SQS:personalize-events-queue.
6
PersonalizeEventLambda calls personalize-events.putEvents() to record the interaction. Personalize model updates in real-time (via Event Tracker).
7
SNS triggers NotificationLambda → SES sends personalised feedback email to student within 90 seconds of submission.

Workflow D — Recommendation Engine Flow

1
Every student action (video_watched, quiz_scored, assignment_submitted, module_completed) is recorded as a Personalize Interaction event with a custom eventValue (e.g., quiz score 0–1).
2
Amazon Personalize trains a USER_PERSONALIZATION recipe (HRNN — Hierarchical RNN) on the Interactions dataset nightly via an automated retraining Lambda triggered by EventBridge.
3
3
When a student completes a module, RecommendationLambda calls personalize-runtime.getRecommendations() with the userId and numResults=5.
4
Response includes ranked list of itemIds (courseModuleIds). Lambda enriches each with DynamoDB metadata (title, duration, difficulty) and returns the top-3 to the student dashboard.
5
For cold-start students (fewer than 5 interactions), the system falls back to a POPULARITY_COUNT recipe that returns globally popular items for the enrolled course category.

Sample Lambda — Grading Engine (Python)

lms-grading-engine / handler.py
import boto3, json, os
from decimal import Decimal

comprehend = boto3.client('comprehend')
dynamodb   = boto3.resource('dynamodb')
sns        = boto3.client('sns')

RUBRIC_CLASSIFIER_ARN = os.environ['COMPREHEND_CLASSIFIER_ARN']
SUBMISSIONS_TABLE     = os.environ['SUBMISSIONS_TABLE']
SNS_TOPIC_ARN         = os.environ['FEEDBACK_TOPIC_ARN']

def lambda_handler(event, context):
    for record in event['Records']:
        body       = json.loads(record['body'])
        sub_id     = body['submissionId']
        student_id = body['studentId']
        text       = body['submissionText']

        # ── 1. Comprehend analysis ─────────────────────────────────
        sentiment  = comprehend.detect_sentiment(Text=text, LanguageCode='en')
        key_phrases= comprehend.detect_key_phrases(Text=text, LanguageCode='en')
        rubric_res = comprehend.classify_document(
            Text=text,
            EndpointArn=RUBRIC_CLASSIFIER_ARN
        )

        # ── 2. Score calculation ───────────────────────────────────
        categories = rubric_res['Classes']
        score      = calculate_score(categories)
        feedback   = generate_feedback(categories, key_phrases['KeyPhrases'])

        # ── 3. Write feedback to DynamoDB ─────────────────────────
        table = dynamodb.Table(SUBMISSIONS_TABLE)
        table.update_item(
            Key={'PK': f'SUBMISSION#{sub_id}', 'SK': body['submittedAt']},
            UpdateExpression='SET aiScore=:s, feedbackJson=:f, gradingStatus=:g',
            ExpressionAttributeValues={
                ':s': Decimal(str(score)),
                ':f': feedback,
                ':g': 'GRADED'
            }
        )

        # ── 4. Notify student ─────────────────────────────────────
        sns.publish(
            TopicArn=SNS_TOPIC_ARN,
            Message=json.dumps({'studentId': student_id, 'submissionId': sub_id, 'score': score}),
            Subject='ASSIGNMENT_GRADED'
        )

def calculate_score(classes):
    weights = {'CRITICAL_THINKING': 0.35, 'ACCURACY': 0.40, 'CLARITY': 0.25}
    return round(sum(c['Score'] * weights.get(c['Name'], 0) for c in classes) * 100, 1)

Chapter Marker Generation Algorithm

After Transcribe completes, the lms-transcript-processor Lambda applies a sliding-window topic detection algorithm:

1
Split transcript into 30-second text windows using the startTime field of each word token.
2
Call comprehend.detect_key_phrases() on each window. Track the set of dominant key phrases per window.
3
When the Jaccard similarity between consecutive windows drops below 0.25 (topic shift), mark the boundary as a chapter start. Minimum chapter length: 3 minutes.
4
Use the top key phrase from each chapter's first window as the auto-generated chapter title.

At-Risk Student Detection Logic

The lms-at-risk-detector Lambda runs every night at 02:00 UTC via EventBridge. It scans StudentProgress records and computes a weighted risk score:

Risk Score Formula
# Risk score = weighted sum, range 0 (safe) to 1 (high-risk)
risk_score = (
    0.30 * days_since_last_login_factor     +  # >7 days = 1.0
    0.25 * missed_deadlines_ratio            +  # assignments past due / total
    0.25 * (1 - avg_quiz_score_normalised)   +  # low scores = higher risk
    0.20 * video_drop_off_rate                  # % videos watched < 30%
)

# Thresholds
if risk_score > 0.75:   alert_level = 'CRITICAL'   # Immediate advisor email
elif risk_score > 0.50: alert_level = 'WARNING'    # Student receives motivational nudge
else:                   alert_level = 'NORMAL'

Encryption Strategy

Data at Rest
All DynamoDB tables encrypted with AWS-managed CMK (SSE-KMS). S3 buckets use SSE-S3 (AES-256) by default; sensitive buckets (submissions, transcripts) use customer-managed CMK with key rotation every 365 days.
Data in Transit
TLS 1.2+ enforced everywhere. CloudFront → Origin uses HTTPS only. ALB uses ACM certificate. S3 bucket policies deny HTTP requests (aws:SecureTransport condition).
Secrets Management
No hardcoded credentials. All Lambda environment variables referencing sensitive values use Secrets Manager ARNs. Lambda execution roles grant secretsmanager:GetSecretValue only for specific secret ARNs.
Token Security
Cognito JWTs stored in memory (not localStorage). Short expiry: AccessToken 1h, IdToken 1h, RefreshToken 30 days. Token rotation on each refresh. CSRF tokens on all state-changing API calls.

Interview Talking Points

Q: Why DynamoDB instead of RDS?

The LMS has 4 dominant access patterns: get student profile by ID, get all modules for a course, get a student's progress on a module, and get all submissions for an assignment. Every pattern maps perfectly to a DynamoDB PK/SK query. The data is non-relational (no complex joins), highly variable in shape (each submission has different AI feedback structures), and must scale to 4,800 concurrent end-of-exam submissions without pre-provisioning. RDS would require an Aurora cluster at ~$300/month minimum; DynamoDB costs $0 when idle and scales linearly. PITR replaces the need for scheduled RDS snapshots.

Q: How does the recommendation system handle cold-start students?

Amazon Personalize's USER_PERSONALIZATION recipe degrades gracefully for users with fewer than 5 interactions. We detect this in the Lambda: if the student has fewer than 5 recorded events, we switch the campaign call to use the POPULARITY_COUNT recipe filtered to the student's enrolled course category. This ensures Day-1 students still get useful recommendations (most popular modules in their track) rather than random content.

Q: Why SQS between the submission endpoint and the grading Lambda?

Three reasons: (1) Decoupling — the API returns 202 immediately; the student isn't blocked waiting for Comprehend. (2) Resilience — if the grading Lambda throws an error, the SQS message becomes visible again after the visibility timeout and retries up to 3 times before going to the DLQ. Without SQS, a failed Lambda = lost submission. (3) Throttle management — Comprehend has service quotas; SQS acts as a buffer so 500 simultaneous submissions don't overwhelm the Comprehend quota.

Q: How do VPC Endpoints improve security?

Without VPC Endpoints, all Lambda calls to S3, DynamoDB, SQS, Comprehend, etc. would traverse the public internet through the NAT Gateway — even though both endpoints are in AWS. VPC Endpoints create a private tunnel within the AWS network fabric. For Gateway Endpoints (S3, DynamoDB), it's free and routes traffic through the VPC routing infrastructure. For Interface Endpoints, traffic never leaves the AWS backbone, which means: no exposure to internet-based man-in-the-middle attacks, the traffic doesn't appear in VPC Flow Logs as external, and we can attach endpoint policies that restrict which S3 buckets or DynamoDB tables can be accessed — a security layer on top of IAM.
