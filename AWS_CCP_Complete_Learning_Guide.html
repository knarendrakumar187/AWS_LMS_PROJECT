🌍 Section 1: What is Cloud Computing?

Simple Explanation

Instead of buying and maintaining your own physical servers, you rent computing power, storage, and services over the internet from AWS. You only pay for what you actually use — just like paying for electricity.

🏠 Daily Analogy: Think of cloud computing like renting an apartment vs. building your own house. You don't buy land, bricks, or tools — you just move in and pay monthly. If you need more rooms, you upgrade. If you leave, you stop paying. No maintenance, no upfront cost.

Deployment Models

Private Cloud
Your own cloud

Used by a single company. Full control. Example: Rackspace on-prem.

Public Cloud
Shared infrastructure

Resources owned by a 3rd party and delivered via internet. Example: AWS, Azure, GCP.

Hybrid Cloud
Best of both

Keep sensitive data on-prem, extend with cloud flexibility.

5 Characteristics of Cloud

  • On-demand self-service — provision instantly, no human needed
  • Broad network access — available from any device, anywhere
  • Multi-tenancy — multiple customers share the same physical hardware securely
  • Rapid elasticity — scale up or down automatically with demand
  • Measured service — pay only for what you use

6 Advantages of Cloud

  • Trade CAPEX for OPEX — no upfront capital expense
  • Massive economies of scale — AWS passes savings to you
  • Stop guessing capacity — auto-scale based on real usage
  • Increase speed & agility — deploy in minutes, not months
  • Stop running data centers — focus on your product
  • Go global in minutes — deploy worldwide with one click

Types of Cloud Computing

IaaS
Infrastructure as a Service

Rent raw compute, networking, storage. You manage OS & apps.
Example: EC2

PaaS
Platform as a Service

Platform manages infra, you deploy code only.
Example: Elastic Beanstalk, Heroku

SaaS
Software as a Service

Fully managed software product you just use.
Example: Gmail, Zoom, Rekognition

AWS Global Infrastructure

Regions
Cluster of data centers

e.g. us-east-1, ap-south-1. Most services are region-scoped.

Availability Zones (AZ)
3–6 AZs per region

Each is one or more isolated data centers with independent power & networking.

Edge Locations
400+ worldwide

Used by CloudFront CDN to cache content close to users.

How to choose a region: Compliance requirements → Proximity to customers → Available services → Pricing

🎯 Exam / Interview Points
  • IaaS, PaaS, SaaS — know real AWS examples for each
  • Min 3 AZs per region; separate from each other for disaster isolation
  • Global services: IAM, Route 53, CloudFront, WAF (not region-scoped)
  • Shared Responsibility Model: AWS = security OF the cloud, Customer = security IN the cloud

🔐 Section 2: IAM — Identity & Access Management

IAM is how you control who can access what in your AWS account. It's a global service — not tied to any region.

🏢 Analogy: IAM is like your company's HR + security badge system. The HR department (root account) creates employees (IAM users), puts them in departments (groups), gives them access cards (policies), and assigns roles to contractors (IAM roles). The security principle: give only the minimum access needed — "least privilege."

Core Components

Users
One person = one user

Maps to a real person. Has a password for Console access. Can belong to multiple groups.

Groups
Collections of users

Groups only contain users (NOT other groups). Assign policies to groups, not individual users.

Policies
JSON permission documents

Define what actions are Allowed/Denied on which Resources. Follow least privilege principle.

Roles
Temporary identity for services

Assigned to AWS services (EC2, Lambda) so they can access other AWS resources securely.

IAM Policy Structure

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow", // or "Deny"
    "Action": "s3:GetObject", // what to allow
    "Resource": "*" // on which resource
  }]
}

How to Access AWS — 3 Ways

Console
Management Console

Web UI — protected by username, password + MFA.

CLI
Command Line

Terminal commands — protected by Access Keys (Access Key ID + Secret).

SDK
Programming Libraries

Code integration for Python, Java, JS, etc. Also uses Access Keys.

Security Best Practices

  • Never use root account for daily tasks — lock it away
  • Enable MFA on root account and all privileged users
  • One physical person = one IAM user
  • Assign users to groups; assign permissions to groups
  • Use IAM Roles for AWS services, not access keys
  • Rotate access keys regularly; never share them
  • Audit using IAM Credentials Report + IAM Access Advisor

Architecture: IAM Flow

How EC2 accesses S3 securely

IAM Role attached
→

S3 Bucket Access ✓

No hardcoded credentials needed. The role is temporary and automatically rotated.

🎯 Exam Points
  • IAM is a Global service — not region-specific
  • Root user should ONLY be used for initial setup and specific root-only tasks
  • MFA = password you know + device you own
  • Access Keys are secret — treat them like passwords, don't share
  • IAM Credentials Report = account-level audit | IAM Access Advisor = user-level (what was last used)

🖥️ Section 3: Amazon EC2 — Elastic Compute Cloud

EC2 lets you rent virtual machines (servers) in the AWS cloud. It's the most fundamental AWS service and the foundation for understanding cloud computing.

🏨 Analogy: EC2 is like renting a hotel room. You choose the room size (instance type), check in when you need it, pay per hour/second, and check out when done. You don't buy the hotel. If you need a bigger room, you upgrade. Different rooms have different prices based on amenities (CPU, RAM, storage).

Naming: m5.2xlarge → m = class, 5 = generation, 2xlarge = size

🏖️ Hotel Analogy for Pricing: On-Demand = walk-in rate | Reserved = pre-book 1–3 years | Spot = bid for empty rooms (might get evicted!) | Dedicated Host = rent the whole building | Savings Plan = pay a daily flat fee

Security Groups = Firewall for EC2

  • Controls inbound/outbound traffic to EC2 instances
  • Only ALLOW rules (no deny rules)
  • All inbound traffic is BLOCKED by default
  • All outbound traffic is ALLOWED by default
  • Can reference by IP or by another security group
  • Locked to a Region/VPC combination

  • Spot Instances are cheapest but can be interrupted — NOT for databases or critical jobs
  • Dedicated Host = physical server for YOU. Dedicated Instances = dedicated hardware but may share with same account
  • If app times out → Security Group issue. If "connection refused" → app not running
  • EC2 User Data = bootstrap script runs at first launch only (as root user)

💾 Section 4: EC2 Instance Storage

EBS — Elastic Block Store
Network USB stick

Persistent block storage attached to ONE EC2 at a time. Tied to a specific AZ. Use snapshots to move across AZs.

EC2 Instance Store
Built-in high-speed SSD

Physically attached to host — ultra-fast I/O. Ephemeral: data lost when instance stops. Good for temp/cache data.

EFS — Elastic File System
Shared network drive

Managed NFS — can be mounted on hundreds of EC2s simultaneously across multiple AZs. Pay per use. Linux only.

FSx
High-performance file systems

FSx for Windows — SMB/NTFS protocol.
FSx for Lustre — HPC, ML workloads, sub-ms latency.

💡 Storage Analogy:
EBS = External USB hard drive (plugged into one computer at a time) | EFS = Google Drive (shared by many computers) | EC2 Instance Store = Built-in SSD (fastest but lost if computer is replaced) | EFS-IA = Rarely used files moved to cheap cold storage automatically

EBS Snapshots

  • Point-in-time backup of an EBS volume
  • Can copy across AZs or Regions
  • EBS Snapshot Archive = 75% cheaper, but 24–72 hours to restore
  • Recycle Bin = protection against accidental deletion (1 day to 1 year retention)

AMI — Amazon Machine Image

AMI = pre-configured snapshot of an EC2 instance. Like a golden template you can launch again and again.

Launch EC2
→
Customize
→
Stop Instance
→

🎯 Exam Points
  • EBS is AZ-locked — to move it, take a snapshot first
  • EC2 Instance Store = highest I/O performance but data is lost on stop/terminate
  • EFS = works with Linux EC2 instances only; EFS-IA is up to 92% cheaper for infrequent files
  • EC2 Image Builder = automates AMI creation, testing, and distribution (free service)

⚖️ Section 5: Elastic Load Balancing & Auto Scaling Groups

Vertical Scaling
Scale Up

Make the instance bigger (t2.micro → t2.large). Limited by hardware. Good for databases.

Horizontal Scaling
Scale Out

Add more instances. Implies distributed systems. The cloud way.

High Availability
Multi-AZ

Run app in at least 2 AZs. Survive a data center failure.

Load Balancer Types

TypeLayerProtocolBest For
Application LB (ALB)Layer 7HTTP/HTTPS/gRPCWeb apps, microservices, HTTP routing
Network LB (NLB)Layer 4TCP/UDPUltra-high performance, millions of req/sec
Gateway LB (GWLB)Layer 3IP PacketsIntrusion detection, 3rd party security appliances

Auto Scaling Groups (ASG)

  • Automatically scale EC2 instances based on load (Scale Out = add; Scale In = remove)
  • Set minimum, desired, and maximum instance counts
  • Automatically register new instances with a load balancer
  • Replace unhealthy instances
  • Cost savings — run only what you need

ASG Scaling Strategies

Dynamic
React to demand

Simple/Step: CPU > 70% → add 2 instances. Target Tracking: keep CPU at 40%.

Predictive
Forecast with ML

Uses ML to predict traffic patterns and pre-provision. Great for time-based patterns.

🎯 Exam Points
  • Scalability = can handle larger loads. Elasticity = auto-scaling based on load. Agility = fast deployment (not scaling).
  • ALB = Layer 7 (HTTP), NLB = Layer 4 (TCP), GWLB = Layer 3 (IP) — know the layers
  • Classic Load Balancer was retired in 2023

🪣 Section 6: Amazon S3 — Simple Storage Service

S3 is infinitely scalable object storage. Think of it as Google Drive for AWS — you store files (objects) in buckets (directories).

📦 Analogy: S3 is like a giant warehouse with unlimited storage rooms (buckets). Each item (object) has a label (key/path). You pay only for what you store and transfer. Netflix stores videos here, Nasdaq archives 7 years of financial data in S3 Glacier.

S3 Key Facts

  • Buckets must have globally unique names across ALL accounts and regions
  • Buckets are defined at the region level (not truly global)
  • Max object size: 5TB. For >5GB, use multi-part upload
  • Objects have keys (full path), metadata, tags, and optional version ID

S3 Storage Classes — Cost vs. Access Speed

ClassRetrievalAvailabilityBest For
S3 StandardInstant99.99%Frequently accessed data
S3 Standard-IAInstant99.9%Infrequent access, disaster recovery
S3 One Zone-IAInstant99.5%Non-critical, can recreate data
S3 Intelligent-TieringInstant99.9%Unknown or changing access patterns
Glacier Instant RetrievalMilliseconds99.9%Quarterly access, 90-day min
Glacier Flexible Retrieval1–12 hours99.99%Archives, free bulk retrieval
Glacier Deep Archive12–48 hours99.99%Long-term compliance, cheapest

Durability: All classes have 99.999999999% (11 nines) — if you store 10M objects, you'd lose one every 10,000 years!

S3 Security

  • IAM Policies — for specific IAM users
  • Bucket Policies — JSON rules at bucket level, allow cross-account access
  • Block Public Access — safety net to prevent accidental public exposure (set at account level)
  • Server-Side Encryption (SSE) — enabled by default for all new buckets

Key S3 Features

Versioning
Protect from accidental deletes

Multiple versions of same file. Suspended versioning doesn't delete old versions.

Replication
CRR & SRR

Cross-Region Replication (compliance, latency) & Same-Region Replication (log aggregation). Requires versioning enabled.

Static Website Hosting
Host HTML on S3

S3 can serve static files via a public URL. If 403 error, check bucket policy allows public reads.

S3 Lifecycle Rules
Auto-move between tiers

Move objects to cheaper storage classes automatically based on age.

Snowball & Storage Gateway

AWS Snowball
Physical data migration device

When it takes >1 week to transfer over the internet, use Snowball. Move petabytes offline. Also supports edge computing at remote locations.

Storage Gateway
Hybrid cloud bridge

Connect on-premises systems to S3/EBS/Glacier. Types: File, Volume, Tape Gateway.

🎯 Exam Points
  • S3 looks global but buckets are created in a specific region
  • 11 nines durability = same for ALL storage classes
  • Glacier classes = archiving. Deep Archive = cheapest, 12-48 hours retrieval
  • S3 Intelligent-Tiering = automatically moves objects, no retrieval fees
  • Snowball: use when transfer >1 week over internet; also runs EC2/Lambda at edge

🚀 Section 9: Deploying & Managing Infrastructure at Scale

CloudFormation
Infrastructure as Code (IaC)

Define ALL AWS resources in YAML/JSON templates. Repeatable, version-controlled. Free service (pay for resources). Generates architecture diagrams automatically.

AWS CDK
IaC with real code

Write infrastructure using Python, JS, Java, .NET. CDK compiles to CloudFormation. Great for Lambda + ECS deployments.

Elastic Beanstalk
PaaS — deploy code easily

Upload your code, Beanstalk handles EC2, ELB, ASG, RDS, monitoring. Free service — pay for underlying resources.

AWS CodePipeline
CI/CD orchestration

Code → Build → Test → Deploy — fully automated. Integrates with CodeCommit, CodeBuild, CodeDeploy, Beanstalk, GitHub.

Developer Tools

CodeDeploy
Auto deployment

Deploy to EC2 or on-prem servers. Hybrid service.

CodeBuild
Build & test

Compile code, run tests, produce deployable artifacts. Serverless, pay per minute.

CodeArtifact
Artifact/package manager

Store npm, pip, Maven dependencies. Secure, scalable.

AWS Systems Manager (SSM)

  • Manage EC2 + on-premises servers at scale (hybrid)
  • SSM Session Manager = connect to EC2 without SSH, no port 22 needed
  • Parameter Store = secure storage for config, passwords, API keys (integrates with KMS)
  • Patch Manager = automate OS patching across all servers
🎯 Exam Points
  • CloudFormation = AWS-only IaC | Beanstalk = PaaS (developer focused, handles infra)
  • CodeCommit was discontinued July 2024 — use GitHub instead
  • SSM Session Manager > SSH — no key pairs, no port 22, full audit trail
  • CodePipeline orchestrates the full CI/CD workflow (Code → Build → Deploy)

🌐 Section 10: Global Infrastructure

Route 53
Managed DNS

Translates domain names to IPs. Routing policies: Simple, Weighted (A/B test), Latency (closest server), Failover (DR).

CloudFront
CDN — Content Delivery Network

Cache content at 400+ Edge Locations globally. Reduces latency. DDoS protection via Shield. Works with S3, ALB, EC2.

S3 Transfer Acceleration
Faster S3 uploads

File goes to nearest Edge Location → private AWS network → S3 bucket. Much faster for cross-continent uploads.

Global Accelerator
Static IP + AWS backbone

2 Anycast IPs. Traffic routes through AWS internal network (60% improvement). Good for TCP/UDP, gaming, IoT. No caching.

Outposts
AWS in your data center

Physical rack AWS installs at your premises. Same APIs as cloud. Low-latency, data residency compliance.

WaveLength + Local Zones
Ultra-low latency

WaveLength = 5G edge (ms latency for mobile). Local Zones = extend AWS region closer to users (gaming, media).

CloudFront vs S3 Cross-Region Replication

CloudFront
Global CDN caching

Files cached at edge for a TTL. Great for static content available everywhere. Files may be slightly stale.

S3 CRR
Real-time replication

Near real-time updates. Read only. Must set up per region. Great for dynamic content in specific regions.

🎯 Exam Points
  • CloudFront = caching CDN (static content) | Global Accelerator = no caching, network routing (dynamic content, static IPs)
  • Route 53 routing: Simple (no health checks), Weighted (A/B), Latency (performance), Failover (DR)
  • Outposts = AWS rack in YOUR data center. You're responsible for physical security.

🔗 Section 11: Cloud Integration (Messaging)

When multiple apps need to communicate, use asynchronous messaging to decouple them — so a spike in one service doesn't crash another.

SQS
Simple Queue Service

Producer sends messages to a queue. Consumer reads and deletes them. Decouples app tiers. Message retention: up to 14 days. No limit on queue size.

SNS
Simple Notification Service

Pub/Sub model: one message → many subscribers (email, Lambda, SQS, HTTP). Topic has up to 12.5M subscriptions. No message retention.

Kinesis
Real-time streaming

Collect, process, and analyze real-time streaming data (clickstreams, IoT). Kinesis Data Streams → Data Firehose → S3/Redshift.

Amazon MQ
Message broker for migration

For on-prem apps using MQTT, AMQP, STOMP protocols. Not as scalable as SQS/SNS but protocol-compatible for migration.

📬 SQS Analogy: SQS is like a mailbox between two offices. The sender drops a letter in the mailbox (queue) without waiting for the receiver. The receiver picks up mail at their own pace. If the receiver is on vacation, mail just piles up safely.
🎯 Exam Points
  • SQS = 1 message, 1 consumer (pull model). SNS = 1 message, many subscribers (push model)
  • Kinesis = real-time big data. SQS = async decoupling.
  • Amazon MQ = for migration of apps already using open messaging protocols (RabbitMQ, ActiveMQ)
  • SQS FIFO = messages in order, exactly once delivery

📊 Section 12: Cloud Monitoring

CloudWatch Metrics
Performance data

Metrics for every AWS service (CPU, Network, etc.). Create dashboards. Default: every 5 mins. Detailed: every 1 min ($).

CloudWatch Alarms
Auto-notifications

Trigger actions when metric crosses threshold: scale EC2, send SNS notification, or stop/start/terminate instance.

CloudWatch Logs
Log aggregation

Collect logs from EC2, Lambda, Beanstalk, ECS, etc. Requires CloudWatch agent on EC2. Real-time monitoring.

EventBridge
Event-driven reactions

React to AWS events (S3 upload, IAM login) or schedule cron jobs. Route events to Lambda, SQS, SNS, etc.

CloudTrail
Audit trail

Every API call in your account is logged. Enabled by default. "Who did what, when?" — for compliance & investigation. Store logs in S3.

AWS X-Ray
Distributed tracing

Trace requests through microservices. Find bottlenecks, pinpoint errors. Visual service map of your architecture.

🎯 Exam Points
  • CloudWatch = performance metrics. CloudTrail = API audit trail. X-Ray = distributed tracing.
  • "Resource was deleted?" → Check CloudTrail first!
  • CloudTrail is enabled by default. CloudWatch logs require an agent on EC2.
  • Billing metrics are stored only in us-east-1 region
  • AWS Health Dashboard = general AWS status | Account Health Dashboard = your resources specifically

🏗️ Section 13: Amazon VPC — Virtual Private Cloud

VPC is your private network inside AWS. Think of it as building your own isolated section of the AWS data center with complete control over networking.

🏘️ Analogy: A VPC is like a gated housing community inside a large city (AWS). You have public streets (public subnets) anyone can see, and private cul-de-sacs (private subnets) only residents can access. The gate (internet gateway) controls who enters from outside.

Core Components

VPC
Private network in AWS

Regional resource. You define your IP range (CIDR).

Subnets
AZ-level network partitions

Public = accessible from internet. Private = not accessible directly.

Internet Gateway (IGW)
Door to the internet

Allows public subnets to communicate with the internet.

NAT Gateway
One-way internet for private subnets

Allows private subnets to access the internet (for updates) but blocks inbound connections. AWS-managed.

Security Groups
Instance-level firewall

Stateful — if you allow inbound, response is auto-allowed. Only ALLOW rules.

Network ACL (NACL)
Subnet-level firewall

Stateless — must set rules for both inbound AND outbound. Can DENY. Applied at subnet level.

Connectivity Options

VPC Peering
Connect 2 VPCs

Private network connection between VPCs (same or different accounts). Not transitive — must connect each pair.

VPC Endpoints
Private access to AWS services

Connect to S3/DynamoDB without going through public internet. More secure, lower latency.

Site-to-Site VPN
On-prem to AWS

Encrypted VPN over public internet. Quick to set up (minutes). Uses Customer Gateway (CGW) + Virtual Private Gateway (VGW).

Direct Connect
Dedicated private line

Physical connection between on-prem and AWS. Private, fast, consistent. Takes 1+ month to provision.

Transit Gateway
Hub for thousands of VPCs

One gateway to connect thousands of VPCs + on-prem networks in a hub-and-spoke model.

PrivateLink
Expose service privately

Expose your service to thousands of VPCs securely, without VPC peering or internet. Requires NLB + ENI.

🎯 Exam Points
  • Security Groups = stateful (remember connections), NACL = stateless (each packet checked)
  • VPC Peering is NOT transitive — A↔B and B↔C doesn't mean A↔C
  • Direct Connect = private, slow to set up (months); Site-to-Site VPN = encrypted, over internet, fast to set up
  • VPC Endpoint Gateway = only for S3 and DynamoDB | Endpoint Interface = all other services
  • Transit Gateway = simplifies complex VPC network topologies (hub and spoke)

🧠 Quick Practice Quiz

Q1: A company wants to host a static website with global low-latency access and automatic scaling. Which combination is best?
✅ S3 hosts the static files, CloudFront caches them at 400+ global edge locations for low latency. Auto-scales, cheap, no servers needed.
Q2: You need to run a database that can handle millions of requests per second with single-digit millisecond latency, with no servers to manage. Which service?
✅ DynamoDB is serverless, handles millions of req/sec, single-digit ms latency, unlimited scale. ElastiCache is a cache (not primary DB), Redshift is for analytics (OLAP).
Q3: A resource was accidentally deleted from AWS. Where should you look first to find out who deleted it?
✅ CloudTrail records ALL API calls — including Delete calls. It tells you WHO made the call, WHEN, and from WHERE. This is the first place to investigate any account activity.
Q4: A startup needs to run a web application. They want to focus on coding and not manage servers, OS patches, or infrastructure. Best option?
✅ Elastic Beanstalk is PaaS — upload your code, it handles EC2, load balancing, auto-scaling, patching. Developers only worry about application code.
Q5: Which EC2 pricing option is MOST cost-effective for a fault-tolerant batch processing job that can be interrupted?
✅ Spot Instances save up to 90% — cheapest option. Perfect for batch jobs that can tolerate interruption. Never use Spot for databases or critical applications that cannot be interrupted.
